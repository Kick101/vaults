## SIEM Use Case Development Lifecycle

![[Pasted image 20250518134606.png | 500]]

| **Stage**                      | **Description**                                                                                   | **Example**                                                                           |
| ------------------------------ | ------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------- |
| **1. Requirements**            | Define the purpose of the use case and what should trigger an alert.                              | Detect brute force attack: alert on 10 failed logins in 4 minutes.                    |
| **2. Data Points**             | Identify where login events occur and the sources of logs.                                        | Windows/Linux servers, VPN, web apps generating login logs.                           |
| **3. Log Validation**          | Ensure logs contain critical fields and are being received correctly across authentication types. | Check logs include user, timestamp, source IP, event type for all login attempts.     |
| **4. Design & Implementation** | Define alerting conditions, aggregation rules, and priority levels.                               | Alert if login failures > 10 in 4 mins from same IP; prioritize admin account alerts. |
| **5. Documentation**           | Document SOPs for analysts, including conditions, priority, and escalation contacts.              | SOP includes alert handling steps and contacts for system owners.                     |
| **6. Onboarding**              | Test the use case in development; fix gaps before deploying in production.                        | Tune alert logic to reduce false positives before production rollout.                 |
| **7. Fine-tuning**             | Continually refine based on feedback and evolving threats.                                        | Update rules, whitelist known behaviors, and add new threat patterns.                 |

## How To Build SIEM Use Cases

- Comprehend your needs, risks, and establish alerts for monitoring all necessary systems accordingly.
- Determine the priority and impact, then map the alert to the kill chain or MITRE framework.
- Establish the Time to Detection (TTD) and Time to Response (TTR) for the alert to assess the SIEM's effectiveness and analysts' performance.
- Create a Standard Operating Procedure (SOP) for managing alerts.
- Outline the process for refining alerts based on SIEM monitoring.
- Develop an Incident Response Plan (IRP) to address true positive incidents.
- Set Service Level Agreements (SLAs) and Operational Level Agreements (OLAs) between teams for handling alerts and following the IRP.
- Implement and maintain an audit process for managing alerts and incident reporting by analysts.
- Create documentation to review the logging status of machines or systems, the basis for creating alerts, and their triggering frequency.
- Establish a knowledge base document for essential information and updates to case management tools.


**When creating an #SOP and documenting alert handling, consider the following:**
- process.name
- process.parent.name
- event.action
- machine where the alert was detected
- user associated with the machine
- user activity within +/- 2 days of the alert's generation


After gathering this information, defenders should engage with the user and examine
the user's machine to analyze system logs, antivirus logs, and proxy logs from the
SIEM for full visibility.
